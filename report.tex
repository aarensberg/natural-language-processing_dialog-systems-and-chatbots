\section{Introduction and Motivation}
This coursework builds a neural chatbot from scratch using PyTorch, starting from a Cornell Movie-Dialogs baseline and extending it with attention, stronger decoding, and pretrained word embeddings. The goal is to improve relevance, coherence, and response diversity while keeping the system reproducible and fully trainable by the student.

\section{Datasets and Preprocessing}
The main dataset is the Cornell Movie-Dialogs Corpus, which is converted into single-turn input-response pairs by traversing conversation chains and pairing each utterance with the next one. Pairs are filtered for length and quality, tokenized with punctuation separation, lowercased, and truncated to fixed maximum lengths. Splits are performed by conversation to reduce leakage across train, validation, and test sets.

\section{Exploratory Data Analysis}
The preprocessing pipeline reports dataset size, vocabulary size, average source and target lengths, token frequency summaries, and noisy examples such as very short utterances or punctuation-heavy pairs. These statistics are exported as JSON and plotted as histograms and token-frequency charts under the artifact directories.

\section{Baseline Model}
The baseline is a word-level GRU encoder-decoder trained with teacher forcing. It uses a learned embedding layer, a unidirectional recurrent encoder, and greedy or beam-search decoding at inference time. This model establishes the Ex1 reference point and provides the comparison target for later improvements.

\section{Improved Model and Decoding}
The improved model replaces the baseline encoder with a multi-layer bi-directional GRU and adds additive attention in the decoder. The decoder conditions on both the previous token and the attention-weighted encoder context. The pipeline also supports three decoding strategies: greedy decoding, beam search with length penalty, and top-k sampling with temperature. This makes it possible to compare deterministic and stochastic generation.

Pretrained word embeddings are optionally loaded from the extracted GloVe vectors. The current setup uses the 2024 wiki-giga embedding file extracted under the data directory, and the loader reports vocabulary coverage before training starts.

\section{Multiple Datasets and Domain Effects}
Exercise 3 is now implemented as a multi-dataset pipeline in \texttt{src/pipeline\_ex3.py}. The current comparison trains the same attention-based seq2seq model on Cornell only and on Cornell plus PersonaChat. To make the domain signal explicit, each source utterance is prefixed with a corpus tag, so the model can condition on whether the example comes from Cornell or PersonaChat without changing the core architecture.

PersonaChat is parsed directly from the extracted raw text files under \texttt{data/personachat/}. Each corpus is split by conversation to reduce leakage, and the pipeline writes separate EDA summaries, token-length plots, sample conversations, and per-domain BLEU/distinct metrics under \texttt{artifacts/exercise3/}.

A local MPS smoke test on capped data confirmed that the pipeline wiring is correct and that both experiment branches start cleanly. The final Colab run completed for the mixed-domain branch as well, producing the following comparison in \texttt{artifacts/exercise3/outputs/comparison.json}: Cornell+PersonaChat achieved train loss 4.3592, validation loss 4.0182, and test loss 4.0382. The domain-specific losses were 4.0711 on Cornell and 3.9641 on PersonaChat, with corresponding perplexities 58.6207 and 52.6746.

On generation quality, the mixed-domain model improved over the Cornell-only control on validation/test loss and overall BLEU, with greedy BLEU 0.0159, beam BLEU 0.0139, and top-k plus temperature BLEU 0.0084. The diversity scores stayed low for greedy and beam decoding but increased markedly for top-k sampling, especially on distinct-2 (0.0989 overall, 0.1119 on Cornell, 0.1286 on PersonaChat). This supports the usual tradeoff: PersonaChat improved the model's domain adaptation and response quality slightly, while stochastic decoding remained the main driver of output diversity.

\section{Memory, Personality, and Feedback}
Exercise 4 is implemented as a lightweight rule-based conversational memory in \texttt{src/memory\_ex4.py} and a comparison pipeline in \texttt{src/pipeline\_ex4.py}. The memory layer stores explicit facts from the dialogue state, including the user's name, location, preference, and recent statements. It then answers direct memory questions such as ``What is my name?'' or ``Where do I live?'' without relying on the seq2seq generator.

This choice is intentionally simple and defensible for the codebase: it is deterministic, easy to inspect, and does not require retraining the neural chatbot. The memory-aware wrapper only augments the model input when a direct recall is not possible, which keeps the baseline model unchanged and makes the effect of memory easy to isolate.

A smoke test on five handcrafted cases covered the required categories: names, locations, preferences, and previous statements. The memory-aware version answered all five correctly, while the baseline model produced generic responses such as ``berger.'', ``oklahoma.'', and ``no.'' on the same prompts. This demonstrates a clear improvement in conversational context handling.

The main limitation is that the mechanism is slot-based and rule-driven, so it handles only the forms it has been programmed to recognize. It will not generalize to arbitrary factual memory, paraphrased references, or conflicting statements unless the rule set is expanded.

Exercise 5 builds on the same idea of explicit control, but shifts from conversation memory to persona consistency. The implemented persona is Ari, a calm chatbot research assistant who likes tea, books, hiking, and clear explanations. Consistency is encouraged by combining three mechanisms: a stable profile description, a persona-prefixed input prompt, and rule-based overrides for direct identity questions and adversarial requests to change the character.

The Exercise 5 smoke test uses six cases, including repeated name and role questions, adversarial attempts to reassign the persona, and self-description prompts. The persona-aware wrapper answered all six cases correctly, while the baseline model remained generic. This indicates that personality consistency is produced primarily by rules plus prompt conditioning, not by the raw generator alone.

Exercise 6 implements a lightweight feedback loop and a persistent `FeedbackStore` that maps normalized user queries to corrected responses. Corrections are stored under `artifacts/exercise6/outputs/feedback_store.json` and are automatically consulted during sample generation and evaluation: when a user query matches a stored key, the corrected response replaces the generator output. A smoke test with five corrections demonstrates reproducible before/after artifacts (`feedback_before.json`, `feedback_after.json`, `feedback_results.json`, and `feedback_report.txt`), showing that user-provided corrections can deterministically override model outputs at inference time.

To make feedback usable in deployed scenarios, an HTTP inference wrapper (`src/server_infer.py`) exposes `/infer`, `/add_correction`, and `/feedback` endpoints so a UI or a human annotator can interactively query the model and store corrections. We also extended the test harness to run paraphrase/generalization checks: the pipeline writes `generalization_results.json` and `generalization_report.txt` that measure how many paraphrased queries are matched by stored corrections. The lookup now uses a lightweight semantic score combining character similarity and token overlap, so paraphrases can resolve even when the wording changes substantially. This gives a simple coverage metric for the persistent feedback store.

For the larger Exercise 6 evaluation, we ran a dedicated protocol in `src/evaluate_feedback_ex6.py` over 23 cases, including 17 positive paraphrases and 6 negative controls. On the current run the system reached positive coverage 0.882, false-positive rate 0.167, and overall accuracy 0.870. The errors were concentrated in the self-description cases, where the semantic matcher over-generalised to the name correction or failed to resolve the paraphrase, and in one unrelated control query that triggered a false positive. The protocol writes `large_eval_results.json`, `large_eval_report.txt`, `large_eval_before_after.txt`, and `large_eval_failure_cases.json`, which provide the required before/after examples and an explicit failure analysis.

\section{Evaluation and Results}
Two runs were executed to validate the improved pipeline. A smoke test on 100 conversations confirmed that the attention-based model, GloVe loading, and the three decoding paths all execute end-to-end. With the extracted GloVe file, coverage reached 89.6\% on the smoke vocabulary and 93.0\% on the full Cornell vocabulary. The smoke run produced the following metrics: train loss 4.4823, validation loss 3.1685, test loss 3.4739, with corresponding perplexities 88.4359, 23.7708, and 32.2609.

The qualitative samples show the expected limitations of a short run: greedy and beam outputs remain generic, while top-k sampling is slightly more varied but still sparse. A later Colab run on the full Cornell corpus used 5 epochs, batch size 128, CUDA, and the extracted GloVe vectors. It produced train loss 4.3799, validation loss 4.0673, test loss 4.0786, with perplexities 79.8270, 58.3982, and 59.0631. The generation metrics were also computed per decoding strategy: greedy BLEU 0.0059 with distinct-1 0.0028 and distinct-2 0.0110; beam BLEU 0.0054 with distinct-1 0.0040 and distinct-2 0.0137; top-k plus temperature BLEU 0.0052 with distinct-1 0.0110 and distinct-2 0.1079. The generated artifacts are stored under [artifacts/exercise2/outputs](artifacts/exercise2/outputs).

A matched no-GloVe run with the same budget was also executed for comparison. It produced train loss 4.3422, validation loss 4.1368, test loss 4.1441, with perplexities 76.8777, 62.6046, and 63.0630. Its generation metrics were greedy BLEU 0.0076 with distinct-1 0.0031 and distinct-2 0.0130; beam BLEU 0.0059 with distinct-1 0.0052 and distinct-2 0.0180; top-k plus temperature BLEU 0.0050 with distinct-1 0.0118 and distinct-2 0.1079. This comparison suggests that GloVe improves validation and test loss, while the no-GloVe run is slightly stronger on greedy and beam BLEU.

\section{Ablation Study}
The current implementation supports ablations by toggling GloVe initialization, freezing embeddings, changing the number of layers, and comparing decoding strategies. A useful ablation for the final report is to compare the full model against a no-GloVe variant and against greedy decoding only.

\section{Error Analysis}
The smoke samples already reveal typical chatbot failure modes: repeated \texttt{<unk>} tokens, empty or nearly empty responses, over-reliance on punctuation, and weak semantic alignment with the input. These failures are consistent with a small training budget and a limited vocabulary built from filtered dialogue pairs. Even in the better Colab run, the BLEU scores remain very low, which confirms that the model still produces mostly generic dialogue despite improved losses.

\section{Limitations, Ethics, and Future Work}
The current system is still a word-level seq2seq model and therefore cannot maintain long-term context or robust factual grounding. Movie dialogue also induces stylistic bias and dramatic responses. Future work should add memory, persona control, feedback storage, and stronger evaluation metrics such as BLEU and distinct-n to better characterize quality and diversity.

\section{Conclusion}
The project now includes a reproducible PyTorch baseline, an improved attention-based chatbot, pretrained embedding support, multiple decoding strategies, a dedicated Exercise 3 pipeline for domain-effect analysis, a deterministic Exercise 4 memory layer, and an Exercise 5 persona-consistency wrapper. Exercise 3 is complete with final Cornell-only and Cornell+PersonaChat results, Exercise 4 shows that a small rule-based memory can recover simple facts reliably, and Exercise 5 shows that a stable persona can be enforced consistently with rules plus prompt conditioning.
